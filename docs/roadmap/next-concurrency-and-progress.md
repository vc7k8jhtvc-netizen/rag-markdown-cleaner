# 下一版本并发与进度路线

状态：规划中

依据：《RAG 清洗工具：下一版本功能与架构构想》（2026-07-24）

## 目标与当前边界

当前版本支持多文件并发，但单个文件的分片仍按编号串行处理。即使存在空闲并发槽，
同一文件的后续分片也不会并行请求 API。

下一版本的核心目标是保留 `workers` 这个唯一的用户参数，并将其定义为全局 API 任务
并发上限。最终调度器可以在不同文件及同一文件的不同分片之间分配任务槽，但任何时刻
实际 API 请求数都不得超过 `workers`。

在下一版本完成前，当前版本继续保持“多文件并发、单文件分片串行”。本路线本身不改变
运行逻辑、版本号、产品行为、发布状态或分支合并状态。

## 分阶段范围

| 阶段 | 任务 | 风险模式 | 范围与交付物 |
| --- | --- | --- | --- |
| P0.1 | 自动续跑 PowerShell 行为测试 | 标准模式，测试补齐 | 锁定既定自动续跑语义；只补测试与测试夹具，不改变产品行为。独立任务见 [P0.1 任务说明](../tasks/p0.1-powershell-auto-resume-tests.md)。 |
| P0.2 | 运行心跳和 SSE 接收量 | 标准模式，但需 Windows 输出验收 | 在现有调度语义不变的前提下，展示请求持续时间、SSE 原始接收字节数和最近数据时间；交互终端与逐行日志分别验收。 |
| P1.1 | 独立分片执行单元 | 标准偏高风险，重构但行为不变 | 从文件级循环中拆出可独立调用的单分片执行单元，仍串行调用；以等价性测试证明输出、缓存和恢复行为不变。 |
| P1.2 | 固定上限全局分片并发 | 高风险模式，必须独立复核 | 建立全局分片任务池、公平调度、单文件并发控制和一次性有序组装；`workers` 成为固定的全局 API 任务并发上限。 |
| P1.3 | 多任务实时面板 | 标准模式，但需 Windows 输出验收 | 基于单一主线程渲染器展示总体进度、活动任务、分片进度和批次统计；不改变调度。 |
| P2.1 | 动态并发调节 | 高风险模式，涉及 API 成本和限流行为 | 根据 429、超时、可重试服务错误与稳定成功情况调节有效并发；范围始终为 1 到用户设置的 `workers`。 |
| P2.2 | ETA | 标准模式，展示增强，不影响调度 | 样本充分且估算稳定时展示预计剩余时间；ETA 只用于显示，不反馈到调度决策。 |

P0 先锁定既定行为并改善运行可见性；P1 完成固定上限的核心并发架构；P2 仅在固定
并发稳定后增加自适应和预测信息。不得把 P0.2、P1 或 P2 功能夹带进 P0.1。

## 自动续跑产品决策

“开始处理全部文件”继续自动判断是恢复旧批次还是新建批次，不要求用户手工选择：

1. 最新批次存在 `pending`、`interrupted` 或 `running` 文件时，继续该批次。
2. `failed` 是原批次终态；仅剩失败项时不直接触发 `--resume-batch`，下一次全量扫描
   通过缓存校验跳过有效分片，并重新处理缺失或失败分片。
3. 不存在可恢复任务时才创建新的全量批次。
4. 恢复严格保留原批次范围；部分文件选择批次完成后不隐式启动全量批次。
5. 已完成且校验有效的分片必须复用，避免重复 API 调用和额外费用。
6. 自动恢复时明确显示“检测到上次未完成任务，正在自动继续...”。
7. 菜单不增加“继续旧任务 / 新建全量任务”的判断步骤，也无需改名为“智能开始”。

P0.1 只通过 PowerShell 行为测试锁定上述决策，不改菜单或 Python 运行逻辑。

## 必须保留的安全能力

- 分片 metadata 与 SHA-256 校验，以及分片结果原子写入。
- 停电、断网或意外关闭后的断点续跑。
- 有效缓存自动跳过，失败分片可以独立重试。
- `workers` 的全局 API 并发上限与共享 429 冷却。
- 暂停或安全停止后不再提交新任务，并持久化未完成状态。
- 最终文档仅组装一次，且严格按分片编号排序。
- 旧批次 manifest、旧分片缓存及现有恢复语义的兼容性。
- 输入路径安全校验、源文件哈希保护、日志隐私与密钥隔离。

测试与验收只能使用临时目录、伪造配置和 mock API；不得读取真实 `.env`、API Key、
Token 或真实业务文件。

## 高风险点

- 同一文件多个分片并发失败时，失败阈值、文件终态和后续分片策略可能产生竞态。
- 暂停、安全停止或进程异常时，已提交任务与未提交任务的状态边界必须可恢复。
- 全局任务槽、共享 429 冷却和动态调节叠加后，可能超限、扩大 API 成本或造成饥饿。
- 多线程更新 manifest、缓存和文件状态可能损坏恢复数据或重复提交 API 请求。
- 多个分片同时完成可能触发重复组装，或在分片未齐时提前发布结果。
- 并发结果必须有序组装，并继续验证源文件与分片哈希，避免发布过期结果。
- 旧 manifest/schema 与旧缓存必须有明确兼容或保守拒绝策略。
- 实时面板在 PowerShell 5.1、Windows Terminal、窄终端和重定向输出中可能错行、
  重叠或泄漏 ANSI 控制字符。
- SSE 接收量仅表示响应体网络活动，不是正文字符数、完成百分比或 ETA 依据。

P1.2 和 P2.1 必须在独立开发分支实施，并由未参与主要实现的人独立复核调度、成本、
限流、恢复和停止语义。

## 阶段验收标准

### P0.1 自动续跑测试

- PowerShell 行为测试覆盖 latest/manifest 缺失、损坏、schema 不兼容，以及包含
  `pending`、`interrupted`、`running` 或仅 `failed` 的最新批次。
- 覆盖全量批次、部分选择批次、命令参数、用户提示和项目 `.venv` 不可用场景。
- 测试不访问网络、真实密钥、真实 `.env` 或真实业务文件，并可在 Windows CI 或明确的
  Windows 验收环境中重复运行。
- 现有产品行为、菜单文案契约和 Python 运行逻辑不变；现有测试继续通过。

### P0.2 运行心跳和 SSE 接收量

- 每个请求可报告开始时间、已读取 SSE 原始字节数、持续时间和最近数据时间，更新经过节流。
- 非流式请求显示等待心跳；接收量不参与完成率或 ETA 计算。
- 所有输出仍由主线程集中渲染；工作线程不直接写 `stdout` 或 `stderr`。
- PowerShell 5.1、Windows Terminal、重定向输出和无交互 CI 均完成专项验收；降级日志无
  ANSI 控制字符且不重叠。
- manifest、缓存、重试、恢复与“多文件并发、单文件分片串行”行为不变。

### P1.1 独立分片执行单元

- 单分片任务单元覆盖请求、重试、质量检查、partial 保存、metadata、哈希与原子写入。
- 调用次序仍为单文件分片串行，日志、输出字节、缓存命中、失败统计和组装结果与重构前等价。
- 全部分片成功并校验后才组装；完整回归、恢复和故障注入测试通过。

### P1.2 固定上限全局分片并发

- `workers` 是唯一并发参数，任意时刻实际 API 请求数均不超过该固定上限。
- 优先跨文件分配任务，剩余槽位公平轮询；存在其他文件时执行单文件上限，无其他文件时
  单文件可使用空闲槽位；有效缓存不占 API 槽。
- 暂停、停止、崩溃、429、分片失败、重试和恢复场景均不丢状态、不重复有效请求。
- 同一文件只组装一次，且仅在全部分片成功校验后按编号组装。
- 新旧 manifest 与缓存兼容测试、并发压力测试、故障注入测试和独立复核全部通过。

### P1.3 多任务实时面板

- 显示文件/分片总体进度、活动任务数、每个任务状态和批次成功/失败统计。
- 活动任务使用稳定 ID 和固定行；长文件名、窄终端、终端尺寸变化均不重叠。
- 普通日志与动态区域协调输出；重定向和 CI 自动使用节流后的逐行格式。
- 完成 PowerShell 5.1、Windows Terminal 和输出重定向验收；面板不改变调度与持久化语义。

### P2.1 动态并发调节

- 429、超时、可重试服务错误、稳定成功的阈值、观察窗口和恢复速度均有明确规格与测试。
- 有效并发始终在 1 到 `workers` 之间；不修改用户设置、缓存身份或批次恢复语义。
- 永久配置错误不会错误触发调节；`Retry-After` 与共享冷却严格生效。
- 使用 mock/受控服务验证 API 请求上限与调用数量，并独立复核成本及限流风险。

### P2.2 ETA

- 仅在样本数量和稳定性满足约定阈值时显示，信息不足或波动过大时隐藏。
- 正确处理缓存命中、并发变化和文件大小差异；失败与重试不会被误报为完成。
- ETA 不影响任务排序、并发、重试、暂停、停止或恢复行为。

## 实施门禁

每项任务使用独立分支和独立 PR，按 P0.1、P0.2、P1.1、P1.2、P1.3、P2.1、P2.2
顺序推进。上一阶段的验收标准未满足时，不进入依赖它的后续阶段。尤其是 P1.2 必须建立
在 P1.1 的行为等价性之上，P2.1 必须建立在固定并发稳定和完成独立复核之后。
